# 上下文与记忆管理

> 解决"记忆问题"：如何在窗口寿命有限、且会被压缩的前提下，不丢失开发状态。

## 按阶段分窗口

规划、执行、审查需要的上下文不是同一批。规划阶段积累的探索、比较、
弯路，对执行阶段是**噪音**，会稀释注意力、诱使模型重新纠结已拍板的决定。

- 每个阶段尽量开新窗口。
- 触发换窗口的信号不是"上下文到了 X%"，而是**任务性质变了**（规划→执行）。
- 换窗口很廉价——前提是重要信息已经落盘。

## STATE 账本：agent 的持久记忆

长任务必须维护一个磁盘上的状态账本（见 `templates/STATE.template.md`）。
每完成一步就更新它，记录：

- 已完成什么
- 下一步是什么
- 踩到的坑、做过的决定

有了账本，上下文被压缩得干干净净、进程崩了、或开一个全新窗口，
都能靠读账本立刻知道自己在哪。**账本才是记忆，窗口只是草稿纸。**

## 早点刷盘

不要等窗口到 80% 才想起来存。plan、关键决策一产生就落文件。
舍不得换窗口的焦虑，几乎都源于"外化"没做够。

## 手写交接 优于 自动压缩

主动写一份自包含的交接物（plan / STATE），让新窗口读，
远胜于依赖工具的 auto-compact：

- 手写交接：**你**控制保留什么。
- 自动压缩：有损，且你**不控制**它保留什么。

## 自包含 plan 的检验标准

一份 implementation plan 好不好，看它能否让一个**对项目一无所知、
没参与过讨论**的新窗口照着把活干对。结构见 `templates/PLAN.template.md`。

## 精细活控制在 50% 以内

不是硬规则：做需要精度的工作时，尽量让有效上下文别超过一半，
避免退化影响输出质量。
