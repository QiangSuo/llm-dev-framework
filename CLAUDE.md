# 共享核心方法论

# 总纲：大模型开发的核心原则

> 这是工具无关的方法论核心。所有具体规则都从这里的心智模型推导而来。

## 核心心智模型：对话窗口是 RAM，磁盘是硬盘

对话窗口（context）不是模型的记忆，它是**易失的工作内存（RAM）**：

- 它有限，且会随着填满而退化（注意力被摊薄、早期内容被有损压缩）。
- 它随对话结束、崩溃、压缩而消失。
- 依赖它保存重要信息，等于把数据只存在 RAM 里不落盘。

真正的记忆是**文件系统（硬盘）**：plan、决策、进度、账本都该落成文件。

由此推出一条贯穿全框架的铁律：

> **任何重要的东西，都必须在它离开窗口之前，已经写进磁盘。**

做到了这一点，"换对话"和"上下文被压缩"就都变成廉价、可生还的操作——
因为窗口里从不存在"只活在窗口里"的关键信息。

## 三条总原则

1. **状态外化（Externalize state）**
   开发的真实状态活在磁盘上（代码、plan、STATE 账本、日志），不活在对话里。

2. **小步可验证（Small verifiable steps）**
   把工作切成小到能在一个上下文预算内干净做完的步骤，每步都有明确的验证。

3. **从产物重新定位（Re-derive position from artifacts）**
   要知道"我在哪"，靠读磁盘上的真实状态（git、测试、文件），而不是靠回忆对话。

## 两类失败，两套药

- **记忆问题**（失忆、丢上下文）→ 用状态外化 + 重新定位解决。见 `10-context-and-memory.md`。
- **行为问题**（跑偏、空转、假装成功）→ 用卡住协议 + 边界 + 验证诚信解决。见 `20-long-horizon.md`。


# 上下文与记忆管理

> 解决"记忆问题"：如何在窗口寿命有限、且会被压缩的前提下，不丢失开发状态。

## 按阶段分窗口

规划、执行、审查需要的上下文不是同一批。规划阶段积累的探索、比较、
弯路，对执行阶段是**噪音**，会稀释注意力、诱使模型重新纠结已拍板的决定。

- 每个阶段尽量开新窗口。
- 触发换窗口的信号不是"上下文到了 X%"，而是**任务性质变了**（规划→执行）。
- 换窗口很廉价——前提是重要信息已经落盘。

## STATE 账本：agent 的持久记忆

长任务必须维护一个磁盘上的状态账本（见 `templates/STATE.template.md`）。
每完成一步就更新它，记录：

- 已完成什么
- 下一步是什么
- 踩到的坑、做过的决定

有了账本，上下文被压缩得干干净净、进程崩了、或开一个全新窗口，
都能靠读账本立刻知道自己在哪。**账本才是记忆，窗口只是草稿纸。**

## 早点刷盘

不要等窗口到 80% 才想起来存。plan、关键决策一产生就落文件。
舍不得换窗口的焦虑，几乎都源于"外化"没做够。

## 手写交接 优于 自动压缩

主动写一份自包含的交接物（plan / STATE），让新窗口读，
远胜于依赖工具的 auto-compact：

- 手写交接：**你**控制保留什么。
- 自动压缩：有损，且你**不控制**它保留什么。

## 自包含 plan 的检验标准

一份 implementation plan 好不好，看它能否让一个**对项目一无所知、
没参与过讨论**的新窗口照着把活干对。结构见 `templates/PLAN.template.md`。

## 精细活控制在 50% 以内

不是硬规则：做需要精度的工作时，尽量让有效上下文别超过一半，
避免退化影响输出质量。


# 长程自主执行

> 解决"行为问题"：长时间无人看管时，如何不跑偏、不空转、不假装成功。

## 重新定位协议（每个工作循环的第一个动作）

不管是刚开始、还是刚被压缩过，干活前**先重新定位**：

1. 读 `PLAN.md`（要做什么）和 `STATE.md`（做到哪了）。
2. **核对磁盘真实状态**：`git status` / `git log`、跑测试、看文件列表。
3. 从**产物**推导"我现在在哪"，而不是从记忆回忆。

发现记忆与磁盘不符时，以磁盘为准。

## 卡住协议（长任务的头号杀手）

长任务最常见的死法是**空转**：同一个错误改一点、再改一点、换个花样再改，
把上下文烧光却离解决更远。硬性规则：

- **同一问题失败 2 次，立即停止增量小修补。**
- 停下来二选一：
  - **退一步诊断根因**，换一个根本不同的思路；或
  - 记进 STATE，**停下来问人**。
- 判断标准："我是在解决问题，还是在换着法子撞同一堵墙？"

## 自主边界：必须先问、不得擅自执行的动作

无人值守时爆炸半径的意义完全不同。以下动作**必须先获得人的确认**：

- 删除数据 / 删除多个文件 / 递归删除
- `git push --force`、`git reset --hard`、`git clean -f`、`branch -D`
- 修改生产环境、部署
- 改动认证 / 授权 / 访问控制
- 安装可疑或来源不明的依赖
- 向外部服务发送代码、密钥或用户数据

边界内放手跑，碰到边界就停下等人。清晰的边界是"敢让它长跑"的前提。

## git 检查点：可回滚性

STATE 账本记"我在哪"，git 记"代码回得去哪"。

- 每完成一个可验证的小步就 **commit 一次**，message 说清这步干了啥。
- 任何一步搞砸，可干净回退到上一个 good 状态。
- 只做新 commit；`push` / `force` / `reset` 属于自主边界，留给人。

## 验证诚信：假绿比没有更危险

每步收尾要过验证闸门（测试通过 / 能编译 / 文件存在）。但要防作弊：

- **禁止**通过删除断言、mock 掉一切、硬编码返回值、skip 测试来骗取绿灯。
- 测试必须真的验证行为。改动后若测试变简单了，要说明为什么。
- 理想做法：**先写测试再写实现**，让目标先钉死，实现只能去够它。

> 一个能被糊弄的闸门，比没有闸门更坏——它给你虚假的安全感。

## 范围纪律：只做 plan 里的事

长时间自主跑，模型天然会漂（顺手重构、加没要的功能、过度设计）。

- **只做 PLAN 里的事**；想多做的先记进 STATE 问人，不擅自扩大范围。
- 但别死守过时的 plan：执行中发现 plan 本身错了（现实与假设不符），
  应**更新 plan 并注明原因**——既不闷头照错做，也不偷偷改道。plan 是活的，改动留痕。

## 审计日志：让人能事后复盘

STATE 朝前看（下一步干嘛），JOURNAL 朝后看（发生过什么）。
见 `templates/JOURNAL.template.md`。记录跑了一程都做了什么、
在哪些岔路口怎么选、为什么。让人回来一扫就知道该信任什么、该复查什么。

## 完成定义

每个任务要有明确的"做完长什么样"（验收标准），否则 agent 不知道何时该停。
